# 기간 이벤트 시스템 설계서 (Period Event System)

> **작성자**: 윤승종  
> **작성일**: 2026-06-16  

## 1. 개요
이벤트의 활성화 및 동작 여부가 특정 날짜 및 시간(Start Date ~ End Date) 범위 내에 묶여 있는 시스템입니다. 기간이 만료되면 더 이상 이벤트 진행도를 올릴 수 없고 보상 획득도 제한됩니다.

## 2. 주요 클래스 및 책임
*   **`ITimeProvider` & `DefaultTimeProvider`**
    *   시스템의 현재 시간을 가져오는 인터페이스입니다. 시간 검증 로직에서 `DateTime.Now`를 직접 호출하지 않고 Provider를 거쳐 시간 조작 및 단위 테스트를 가능하게 합니다.
*   **`EventDefinitionDTO` (Model Data)**
    *   해당 이벤트가 기간 제한이 있는지 판별하는 플래그(`isTimeLimited`)와 시작/종료 일시 데이터를 보유합니다.
*   **ViewModel 계층 (`EventListViewModel`, `EventDetailViewModel`)**
    *   `ITimeProvider`를 주입받아, 이벤트가 현재 시간 기준으로 유효한지 검사한 후 UI에 노출 여부를 결정합니다. 만료 시 회색 처리 또는 목록에서 제외하는 로직을 수행합니다.

## 3. 동작 흐름 (Data Flow)
1.  **데이터 로드**: `EventModel`에서 모든 이벤트 데이터를 로딩할 때 시작 및 종료 시간을 함께 파싱합니다.
2.  **조건부 렌더링**: `EventListViewModel`은 목록을 갱신하기 전에 각 이벤트의 기간 유효성을 검사합니다.
3.  **행동 차단 (Time Constraint)**: 기간이 만료된 이벤트는 디버그나 인게임 액션 트리거가 발생하더라도 `CanAddProgress()` 진입 시 거부(Reject)되어 진행도가 멈춥니다.
4.  **시간 시뮬레이션**: 디버그 UI의 '가짜 시간 주입' 버튼을 통해 미래/과거 시간을 강제로 설정(`SimulateTimeOffset`)하면 전체 기간 이벤트들이 즉각적으로 갱신되어 UI가 동기화됩니다.

## 4. 확장성 및 계층형 구조 (Hierarchy)
*   기간(Period) 메타데이터는 **최상위 이벤트(Event)** 계층이 독점하여 관리합니다.
*   하위의 개별 퀘스트인 **퀘스트(Quest)**들은 이벤트의 기간 유효성 검증 결과를 상속받습니다. 이벤트가 만료되면 산하의 모든 퀘스트 진행이 일괄 차단되므로, 여러 개의 복합 퀘스트(출석, 킬수 등)를 하나의 기간 이벤트 그룹으로 묶어 관리하기 용이하도록 설계되었습니다.
