## 1. README 작성 필수 원칙

1. **상용급 문서화 품질**: 외부 개발자나 기획자, QA 등 협업하는 모든 인원이 이 문서 하나만 보고도 프로젝트의 핵심 아키텍처를 파악하고, 로컬 환경에서 빌드 및 테스트를 수행할 수 있어야 합니다.
2. **기술 스택 및 설계 원칙 명시**: 단순히 개발 환경만 나열하는 것이 아니라, 사용된 핵심 디자인 패턴(Pure DI, MVVM, Strategy, Factory 등)과 그 의사결정 이유를 명확하게 밝혀야 합니다.
3. **폴더 및 파일 구조 시각화**: 주요 코드와 리소스가 어디에 분산되어 있는지 파악할 수 있도록, 디렉토리 구조 트리와 설명이 포함되어야 합니다.
4. **한글 작성**: 모든 README 내용은 한국어를 기반으로 친절하고 격식 있는 어조로 작성합니다.

## 2. 시스템 설명
### 2.1 전체 시스템 구조 설명
- 아키텍처의 의존성 흐름, 레이어 구성도(다이어그램 사용).
- 외부 DI 프레임워크가 배제된 상태에서의 객체 라이프사이클 관리 명세.

### 2.2 주요 클래스 역할 설명
아래 클래스들의 책임을 하나씩 명확하게 명세합니다:
- **EventManager (EventModel)**: 전체 이벤트 데이터의 로드, 진행 및 보상 상태를 제어하는 중재 모델.
- **EventTracker (IEventCondition)**: 각 이벤트 인스턴스의 달성 조건을 관리하고 진행도를 추적하는 구현체.
- **RewardSystem (IEventReward / RewardFactory)**: 보상 지급 처리 및 팩토리 시스템.
- **AttendanceEvent (AttendanceCondition)**: 일일 출석 및 누적 출석 추적 로직.
- **MissionEvent (KillCountCondition, StageClearCondition 등)**: 적 처치, 스테이지 클리어 등의 누적 행동 이벤트 추적 로직.
- **EventPointSystem (EventPointManager)**: 스테이지 클리어 및 광고 시청 등을 통해 포인트를 획득하고 보상과 교환하는 시스템.
- **SaveSystem (JsonSaveSystem)**: 로컬 파일 I/O 기반 데이터 영속성 처리기.
- **EventUI (EventListView, EventDetailView 등)**: MVVM 패턴을 따르는 뷰 컴포넌트군.

## 3. 세부 스타일 가이드

### 3.1 어조와 포맷
- 경어체를 사용하며, 개발 용어는 한국어 공식 명칭 또는 영어 단어를 적절히 병행하여 사용합니다. (예: 의존성 주입 또는 DI)
- 파일 경로와 코드 컴포넌트는 반드시 마크다운 코드 블록(`code`) 또는 굵은 글씨로 하이라이트하여 문서 가독성을 극대화합니다.
- 복잡한 데이터 흐름이나 의존성 그래프가 필요한 경우, ASCII 다이어그램 또는 **Mermaid 다이어그램**을 사용하여 시각화합니다.

### 3.2 문서 유지보수 규칙
- 코드의 구조가 수정되거나 신규 기능(새로운 조건/보상 타입 등)이 설계상 확장될 때마다 README.md의 '디렉토리 구조' 및 '아키텍처 명세' 부분을 동기화하여 수정해야 합니다.
- 설계 규칙(`.agent/rules/`)의 변경이 발생한 경우, 연관된 루트 README.md 역시 함께 업데이트하여 일관성을 맞춥니다.
